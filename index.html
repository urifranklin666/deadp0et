deadp0et protocol v0

Accounts, device keys, and server-assisted encrypted delivery.

This version models the real secure-messaging shape: users sign up, devices generate key material locally, the server stores public keys and opaque message envelopes, and only recipient devices can decrypt message bodies.

end-to-end encrypted envelopes server stores ciphertext only device identity plus signed prekeys

Account

Create a user and generate local device keys.

Directory

The server exposes usernames and public key bundles, never plaintext.

Compose

Fetch the recipient prekey bundle, derive a session secret, and send an encrypted envelope.

Inbox

Decrypt messages addressed to the current account using local private keys.

Protocol blueprint

What the app and server agree on for production.

Identity

Each device creates an ECDH identity key pair and a signed prekey locally. The server stores only the public bundle.

Authentication

Passwords authenticate to the service. Device private keys stay in the browser and should later be hardware-backed when available.

Transport

Messages are sealed with AES-GCM using a shared secret derived from sender ephemeral keys and recipient prekeys.

Server role

The server delivers envelopes, tracks nonces and timestamps, and cannot read subjects or bodies after encryption is applied client-side.

Ready to model account creation and encrypted delivery.
